# 家庭网关改造方案（全屋自动分流 + 分层安全）

> 状态：**设计文档，未实现**。记录决策与落地路线，实现时按此走。
> 关联：`../CLAUDE.md`（架构总纲）。本文聚焦「把 trust-proxy 落成家庭主路由」的形态。

---

## 1. 目标（用户需求，已锁定）

1. **全屋一个 WiFi，连上即用，零客户端配置**：自动墙内直连 / 墙外走代理，不用在每台设备上手动设网关。
2. **安全不牺牲**：核心使命线仍是「识别木马/后门向外部回传机密（exfil / C2）」。
3. **在外也能用**：不在家时也享受墙内墙外自动切换；但**不是所有流量都回远程网关**（省流量），可接受手机装 app 靠规则分流。
4. **高可定制 + 稳定翻墙 + 自带内网穿透**。
5. 宽带：**电信 2000M**（PPPoE）。

### 为什么旧方案（旁路由 + tproxy）不好用
旧 Pi 是**旁路由**：主路由才是 DHCP 服务器，下发的默认网关是主路由自己，设备默认不走 Pi。要生效只能**逐台手动改网关指向 Pi** → 这就是「不方便」的根因。tproxy 本身没问题，问题是流量没被自动引到 Pi。

---

## 2. 核心决策

### 2.1 安全姿势：分层，不是全屋默认拒绝
「连上就自由上网」（需 default-allow）与「Strict 默认拒绝」（未知即拒）在**同一批流量上不可兼得**。全屋 Strict = 每个网站都要先加白，不可用。解法是**按设备分段**，每段用对的控制：

| 网段（VLAN/子网） | 设备 | 姿势 | 出口 |
|---|---|---|---|
| **高价值** | 工作机、NAS、存密钥的机器 | **Strict 默认拒绝** + 进程/来源闸 | 按需 |
| **人用** | 手机、平板、个人电脑 | **Split**：CN 直连 / 境外走云出口 / 其余放行 + 全检测自动处置 | 云出口 |
| **IoT** | 摄像头、音箱、电视 | **Strict 默认拒绝**（只放行厂商固定端点） | 直连 |
| 全段 | — | 监控 + 留痕 + 拦已知恶意 | — |

**「安全不牺牲」在现实里的样子**：把值钱数据放 Strict 段（默认拒绝，未批准的目的地一律出不去）；自由上网的设备放 Split 段，靠**检测 + 情报 + 自动断连**兜底（威胁情报命中、beaconing、DGA/DNS 隧道、大上传外泄）。默认拒绝用在能用的地方（IoT/高价值，目标固定，不影响可用性），检测用在必须自由的地方（人用浏览）。

**Split 的安全边界（诚实）**：可靠拦通用威胁（已知 C2、吵闹 beacon、明显 DGA、大上传、恶意域名），并全量可视/留痕；但对**借合法服务外传**（Telegram/Drive/GitHub/域名前置）、**带 jitter 的 C2**、**低慢外泄**、**HTTPS 内容级**、**全新基础设施**是「检测不保证拦截」。物理定律：给设备自由上网 = 给了它外泄通道；检测抓粗心的，抓不住用心的。**所以真外泄防线靠「把值钱数据放 Strict 段」，不靠 Split 的检测。**

### 2.2 拓扑：x86 盒子当主路由（inline），BE7000 转 AP
- 盒子跑 DHCP 当**主路由**，DHCP 自动把「网关=盒子」下发给每台设备 → **连上就走，零配置**（根治旁路由痛点）。
- BE7000 转 **AP 模式**，只管发 WiFi。
- 代价：盒子是全屋单点 + 吃全部吞吐（x86 撑得住 2G）。

### 2.3 墙外出口：自建
- 你的**云服务器 / k8s WireGuard** 当出口。协议优先 **Reality / AnyTLS / Hysteria2**（抗封锁好），自建 IP 不共享、不像机场 IP 被批量封。
- **稳定翻墙 = 出口 + 协议 + failover**（软件层，trust-proxy 现成的 urltest 自动切换），不看路由器机箱。机场订阅当备份节点。

### 2.4 内网穿透：云当中继（与出口一鱼两吃）
- 电信家宽大概率 **CGNAT（无公网 IPv4）**。用**同一台云 + 同一条 WireGuard**：出去翻墙、进来穿透。
- 方式：WireGuard（云有公网 IP 当中继）或云上自建 **Headscale**（Tailscale 控制面，避开国内官方协调服务器抽风）。
- **待确认**：打电信客服问能否给**公网 IPv4**（很多 PPPoE 线申请就给）。给了 → 家里可直接当 WG server + DDNS，最简单；CGNAT → 走云中继。

### 2.5 在外（手机）
- 手机装标准客户端（sing-box / Clash Meta / Shadowrocket），导入一份 **trust-proxy 生成的客户端配置**：**CN 本地直连**（省流量）、**境外才走云出口**。
- 检测在云出口看到经过的境外流量。

## 3. 硬件方案

### 3.1 主路由盒子
- **首选：复用现有 14 核 / 64G 迷你主机**——对路由是性能过剩，2G 全量深检 + trust-proxy 全栈余量极大。
- **唯一硬门槛：网口 ≥ 2 个（WAN + LAN），最好 2.5G Intel i226-V。**
  - **待确认**：这台机器几个网口、什么 CPU 型号。
  - 若只有 1 个网口：补 **USB3 转 2.5G 网卡**（USB3 跑 2.5G 够 2G），或单口 + VLAN 交换机做 router-on-a-stick。
- **备选（要另买）**：`软路由 N305 四网口 i226`（淘宝），N305 + 4×i226-V 2.5G + 16G + 256G NVMe，约 ¥1500–2200。
- **权衡**：14c 芯片 24h 常开功耗/发热/噪音高于 N100（~10W 无风扇）；不在乎就用现成的，想省电安静将来再上 N100。**路由需专机专用 / 虚拟化隔离**。

### 3.2 WiFi
- 先用 **BE7000 转 AP**（单段够用，零额外花费）。
- 将来要 per-SSID 分段（人用/IoT 分开 SSID→VLAN）→ 加 **VLAN-aware AP**：Omada EAP772（~¥1200）/ UniFi U7 Pro。BE7000 大概率做不了 per-SSID VLAN 打标。

### 3.3 树莓派
- **别当主路由**（ARM 顶不住 2G）。留作：低功耗常开 **DNS/监控探针**、**备份网关**（主路由挂了顶一下），或退役。

## 4. 软件栈

推荐 **PVE（Proxmox）虚拟化**（可定制天花板、互不影响）：
```
物理机（14c64g / N305）
 └─ PVE
     ├─ VM: OpenWrt         → 主路由：PPPoE 拨号 + VLAN + DHCP + 防火墙 + 内网穿透(WG/Tailscale)
     ├─ VM/LXC: Debian      → trust-proxy：检测 + 分段策略 + dashboard + 翻墙(sing-box)
     └─ （余量跑其他 homelab 服务）
```
备选：**裸装 OpenWrt**，trust-proxy 静态二进制直接跑在上面（更简单，少一层）。

**流量路径（Split 段示例）**：
```
设备 ─WiFi→ BE7000(AP) ─LAN→ 盒子[OpenWrt 路由] ─→ trust-proxy 分段策略
    ├─ CN(geosite-cn/geoip-cn)      → direct → 出网
    ├─ 境外(geolocation-!cn)         → proxy 组 → WireGuard/Reality → 云出口 → 出网
    └─ 兜底(Split)                   → proxy 组
                                     ↑ 全程过 detector：情报/beacon/DGA/外泄 → 告警/自动断
```

## 5. trust-proxy 需要新建的功能（gap 清单）

现状是**单一全局姿势**（一份白名单、一个兜底 route→block）。分层家庭网关需要：

| 优先级 | 功能 | 说明 |
|---|---|---|
| **P0** | **Segments（分段抽象）** | 一组 `{名字, source_cidr, 姿势(split\|strict), 白名单/出口}`。网关按来源网段注入各自规则 + 各自兜底（strict→reject；split→CN直连/境外proxy/其余放行）。**这是地基。** |
| **P0** | **可配兜底 / `final` + Split 预设** | 现在兜底写死 block；Split 段要 default-allow 分流。做一个「翻墙分流」profile 预设（geosite-cn→direct / geolocation-!cn→proxy / final→proxy）。 |
| **P1** | **客户端配置/订阅导出** | 给手机 app 一键导入（分流规则 + 云出口）。 |
| **P2** | **LAN-DNS + DNS 查询检测** | 主路由模式下 DNS 经我们 → 全网域名监控 + 把 DNS 隧道/DGA 检测做实（观测查询而非仅连接）。 |
| **P3（已设计，暂缓）** | **clash Rule↔Global 开关** | 见 CLAUDE.md 决策记录；安全floor常在 + 一个 Global 直连开关。用户「不着急实现」。 |

已具备（复用）：规则集导入（geosite/geoip）、代理出口（8 协议 + WG/Tailscale endpoints）、TUN 透明捕获、检测四主线、Profiles、多节点大脑、管理端口豁免 + 模式切换死亡开关。

## 6. 切换 Runbook（最小断网 + 可回滚）

**原则**：搭建/测试阶段**零断网**（盒子当 BE7000 下一台普通设备并行搞完）；只有「切换那一下」短暂断网（约 10–30 分钟）；**BE7000 原样留着当秒级回滚兜底**。

### 6.1 切换前检查清单
- [ ] **光猫模式**：桥接 or 路由/拨号？
  - 路由/拨号模式 → 盒子插上直接 DHCP 拿 IP，**不用 PPPoE**，切换最简单。
  - 桥接模式 → 盒子要 **PPPoE 拨号**，**必须提前拿到电信 PPPoE 账号密码**（BE7000 后台或打电信要）。**没账号密码切换会卡死。**
- [ ] **公网 IPv4**：打电信问能否开（影响内网穿透方式）。
- [ ] **MAC 绑定**：若拨不上，把 BE7000 的 WAN MAC **克隆**到盒子。
- [ ] 盒子已并行装好、测好（翻墙/检测/dashboard/云出口/WG 均通）。
- [ ] BE7000 转 AP 的步骤已备好（关它的 DHCP + WAN 拨号，用 LAN 口接盒子）。
- [ ] 选**深夜/家人不用网**的时段。

### 6.2 切换步骤
1. 光猫 WAN 线：从 BE7000 → 拔到盒子 WAN 口。
2. 盒子按预配置 PPPoE 拨号（或 DHCP 拿 IP）。
3. BE7000 切 AP/桥接：关 DHCP、关 WAN 拨号，LAN 口接盒子 LAN。
4. 设备重连 → 从盒子 DHCP 拿到「网关=盒子」→ 验证上网 + 翻墙 + dashboard。

### 6.3 回滚（出问题时，~2 分钟）
1. 光猫线插回 BE7000 WAN。
2. BE7000 恢复路由模式（DHCP + 拨号）。
3. 家里恢复上网。

## 8. 里程碑（实现时）
1. **M-A 硬件就位**：确认网口 → 装 PVE/OpenWrt → PPPoE/DHCP 上网（并行，不切换）。
2. **M-B trust-proxy Segments + Split**（P0）：分段策略 + 翻墙分流 profile，盒子上跑通。
3. **M-C 云出口 + failover**：自建 Reality/AnyTLS 出口 + urltest；WireGuard 双向（翻墙出 + 穿透进）。
4. **M-D 切换**：按 §6 runbook 切主路由，BE7000 转 AP，验证 + 保留回滚。
5. **M-E 客户端导出**（P1）：手机在外分流。
6. **M-F 分段细化 / LAN-DNS**（P2）：VLAN AP 多 SSID、DNS 查询检测。
